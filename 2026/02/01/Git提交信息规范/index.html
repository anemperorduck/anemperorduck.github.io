Git提交信息规范

发表于2026-02-01|更新于2026-02-02|教程
|浏览量:

Commit 信息（提交说明）不仅仅是写给自己看的，更是写给团队和未来的自己看的。 规范的 Commit 信息可以让项目的开发历史一目了然，方便回溯问题，甚至可以自动生成更新日志（Change Log）。

目前业界最广泛采用的规范是 Conventional Commits (约定式提交)，它主要源于 Angular 团队的规范。

1. 基础格式公式

一个完美的 Commit 信息通常包含三个部分：Header（标题）、Body（正文）、Footer（脚注）。
对于初学者，重点掌握 Header（标题） 即可。

2. Header 的详细规范（核心）

Header 只有一行，包括三个字段：type（必需）、scope（可选）和 subject（必需）。

A. Type（类型）

这是最重要的部分，用来告诉大家这次提交做了什么类别的修改。常见的类型如下：

类型 含义 说明
feat Feature 新增功能（最常用）
fix Fix 修复 Bug（最常用）
docs Documentation 仅修改了文档（如 README.md）
style Style 代码格式调整，不影响逻辑（空格、分号、缩进等）
refactor Refactor 代码重构（既没有新增功能，也没有修复 Bug）
perf Performance 优化性能的代码
test Test 增加测试或修改现有测试
chore Chore 构建过程或辅助工具的变动（如修改 .gitignore, package.json）
revert Revert 回滚上一次提交

B. Scope（作用域 - 可选）

用于说明 commit 影响的范围，比如数据层、控制层、视图层等，视项目而定。

  • 例如：feat(login): ... 表示修改的是登录模块。

C. Subject（简述）

对变更的简短描述。

3. 实例对比

错误的/不规范的写法：

  • git commit -m "update" (更新了什么？)
  • git commit -m "fix bug" (修了哪个bug？)
  • git commit -m "123" (毫无意义)
  • git commit -m "Fixed the login issue and added a new banner image" (一个commit做了两件事，格式冗长)

规范的写法：

  • 新增功能： feat(user): add avatar upload support
  • 修复 Bug： fix(api): handle timeout exception
  • 修改文档： docs: update setup instructions in README
  • 代码重构： refactor: simplify date parsing logic
  • 修改样式： style: remove trailing spaces

4. 进阶：Body 和 Footer（了解）

当你需要解释为什么要这么改，或者怎么解决的复杂问题时，使用 git commit (不带 -m) 进入编辑器模式，编写详细内容。

  • Body（正文）： 详细描述修改的动机。
    • Why? 之前的代码有什么问题？
    • How? 这次修改是如何解决的？
  • Footer（脚注）：
    • 通常用于关联 Issue（任务单）。
    • 例： Closes #123 (GitHub 会自动把这个 commit 和 issue #123 关联并关闭它)。

5. 总结

  1. 原子提交（Atomic Commits）： 尽量一个功能提交一次，或者一个 Bug 修复提交一次。不要写了一整天代码，包含十几个功能的修改，最后只用一个 commit 提交。
  2. 中英文皆可： 如果是个人项目或国内团队，用中文完全没问题（如 feat: 增加登录按钮）。如果是开源项目或国际团队，请使用英文。
  3. 保持统一： 无论你选择哪种规范，最重要的是整个团队（或你自己）前后保持一致。

现在，你可以尝试用下面的格式提交你的下一次代码了：

git commit -m "feat: 完成首页轮播图功能"
