FastAPI 第三节：Agent网关下的参数校验体系——Path、Query、Body

打造一个稳定的 AI Agent 服务，很多人把精力花在模型选型、Prompt 调优和复杂的图编排上，却忽略了一个最基础的环节：入口处的参数校验。当一个大语言模型 API 调用动辄消耗数秒时间和数千 Token 成本时，任由非法请求打到下游 Agent 节点，无异于将钱扔进碎纸机。

这一节我们聚焦 FastAPI 的核心声明系统——Path、Query、Body 与 Field，并将它们放置在企业级 AI 网关的背景下重新理解。你会看到，严格的入参校验不仅是防御手段，更是一种成本控制策略。

参数校验为何重要？

传统的 Web 应用中，参数校验主要是为了安全——防止 SQL 注入、XSS 和格式异常。但在 LLM 驱动的 Agent 架构里，校验失败会直接转化为高昂的 Token 浪费和状态机崩溃。

一个典型的 LangGraph 应用，接收用户请求后会经过条件路由、工具调用、LLM 推理等多个节点。这些节点对初始状态的数据结构高度敏感。如果某个必传字段缺失、temperature 超出合法范围、或者 session_id 的格式不符合 UUID 标准，但在路由入口处没有被拦截，那么：

  • 下游 LangChain 链路可能在中途抛出不可预料的 Pydantic 校验错误，导致整个请求失败，但前面几个节点的 LLM 调用已经消耗了 Token。
  • LangGraph 的 Checkpointer 依赖严格的 thread_id 做状态持久化，格式错误会导致历史对话无法恢复，引发“丢失记忆”的诡异 bug。
  • 流式输出开启与否、温度值范围、历史上下文截断深度等参数，如果没有预检，会让 Agent 的行为变得不可控。

FastAPI 作为 API 门禁层，它的类型系统允许我们在函数体执行之前完成所有入参校验。拒绝非法请求的成本，几乎为零。

核心参数声明体系速览

FastAPI 提供了三套参数声明工具，分别对应 HTTP 协议中的三个请求位点：

1. Path —— 路径参数：负责资源定位

路径参数用于标识资源。在 Agent 服务里，agent_id 和 session_id 是最典型的路径参数。通过 Path(...) 可以为它们附加丰富约束：

  • min_length / max_length：限制字符串长度，防止过长的 ID 攻击。
  • pattern：使用正则表达式限定字符集，比如 ^[a-zA-Z0-9_-]+$ 只允许字母、数字、下划线和连字符。
  • 类型自动转换：将 str 声明为 UUID 后，FastAPI 会在路由层直接将字符串转换成 uuid.UUID 对象。如果字符串不符合 RFC 4122 格式，客户端会立即收到 422 响应，根本不会进入业务逻辑。

这种“先校验后执行”的机制，让 LangGraph 可以安全地使用 thread_id = str(session_id)，而无需在图中再做格式判断。

2. Query —— 查询参数：控制行为策略

查询参数通常用来调整请求的处理方式，而不是传递核心数据。在 AI 场景下，Query 参数的精妙运用能极大提升接口灵活性：

  • temperature：控制 LLM 输出的随机性，取值 0.0~2.0。通过 ge=0.0, le=2.0 限定范围，任何越界调用都会被直接拒绝。
  • max_tokens：限制单次生成的最大 Token 数，避免意外产生巨额费用。
  • verbose：布尔值，控制是否输出调试日志，方便排查问题又不影响生产性能。
  • stream：是否启用流式响应。这个参数一旦写错，客户端可能会错误地解析 JSON 为 SSE 流，严重时导致前端崩溃。

声明 Query 参数时，default、ge、le、gt、lt 等元数据既是校验规则，也是自动生成的 OpenAPI 文档。这对于团队协作和 SDK 生成非常有价值。

3. Body 与 Field —— 请求体字段：承载复杂载荷

现代 AI 应用早已不是简单的“一问一答”。多轮对话历史、附件 URL、工具调用元数据，都需要通过 JSON Body 传递。FastAPI 借助 Pydantic Model 和 Field 函数，把散乱的 JSON 数据转化为强类型对象：

  • min_length / max_length：对用户 Prompt 做长度控制，防止攻击者提交一篇《战争与和平》的请求体，造成 LLM 上下文溢出或 DoS 攻击。
  • pattern：对字符串字段做正则约束，例如消息角色 role 只能取 user、system 或 assistant。
  • default_factory 与 max_length：为历史消息列表设置上限，避免客户端塞入数百条历史记录导致处理超时。
  • examples：提供示例值，直接展示在 Swagger UI 中，降低 API 消费方的使用门槛。

所有这些规则，在 Pydantic v2 的 Rust 内核中以极高性能完成校验。一旦 JSON Body 不满足要求，422 错误会带着明细的失败原因返回，且没有一行 Python 业务代码被执行。

生产环境中的几个关键细节

1. 资源生命周期管理不要偷懒

在路由函数内部创建 LLM 客户端实例，是最常见的反模式。正确做法是利用 FastAPI 的 lifespan 上下文管理器，在服务启动时初始化全局资源（LLM 连接池、数据库连接、Checkpointer），并在关闭时优雅释放。

这样每个请求都可以直接使用 app.state 中的长连接实例，避免反复建连带来的延迟和内存泄漏。

2. 异步路由中使用异步 I/O

路由定义为 async def 后，内部所有可能产生等待的操作都要用 await。如果调用了同步的 requests 库或同步 LLM SDK，事件循环会被阻塞，导致整个进程吞吐量骤降。必要时使用 run_in_threadpool 将同步调用隔离到线程池。

3. 告别 v1 习惯

FastAPI 当前与 Pydantic v2 深度集成。注意 Field 中的正则校验已从 regex 改为 pattern，示例值从 schema_extra 改为 json_schema_extra 或直接使用 examples。继续沿用 v1 写法虽然有时仍被兼容，但会在迁移时埋下隐患。

4. Prompt 注入的基础防护

不要在 Gateway 层对用户 Prompt 做天真拼接。max_length 约束、正则过滤控制字符、以及针对特定关键词的黑名单校验，都可以通过 Pydantic 的自定义 validator 实现。这是成本最低的 Prompt 注入第一道防线。

总结

FastAPI 的参数声明系统远不止是一种语法糖。把它放在 AI Agent 网关的入口处，它就变成了成本控制、稳定性保障和团队协作效率的杠杆。Path 让资源定位变得坚不可摧，Query 让行为策略可配可控，Body 与 Field 则把复杂的载荷约束写在了阳光之下。拒绝一个非法请求，有时比处理它更有价值——尤其是在每一次拒绝背后，都可能是一次 LLM 推理成本的无声节约。
